## 2. RPC 通信框架对比

### 总览

| 维度 | NewLife.Remoting | gRPC | ASP.NET Core Minimal APIs |
|:-----|:----------------|:-----|:-------------------------|
| **传输层** | TCP/UDP/HTTP/WebSocket，可长连接，连接池/单连接 | HTTP/2（必需），双向流/Unary | HTTP/1.1/2/3，REST 友好 |
| **契约** | 消息契约（IMessage/ApiMessage），默认 JSON，可替换 IEncoder | 强契约（.proto），Protobuf | HTTP/JSON，OpenAPI 生态 |
| **序列化** | JSON（默认）+ IPacket 二进制直通 | Protobuf（高性能、跨语言） | JSON |
| **流式调用** | Server-Streaming（IAsyncEnumerable\<T\>），HTTP 兼容 SSE | 双向流/服务端流/客户端流/Unary | 需 WebSocket/SignalR |
| **连接复用** | 长连接 Multiplex 复用，服务端可主动推送单向消息 | HTTP/2 多路复用 | 短连接或 HTTP/2 复用 |
| **元数据传递** | client.Headers 字典透传 TraceId/TenantId | Metadata | HTTP Headers |
| **认证** | Token 注入 + IApiHandler 统一校验 + IRetryPolicy 401 自动重登 | Interceptor + Metadata + TLS | Middleware + Auth |
| **可观测性** | ILog/ITracer/ICounter + 慢调用日志 + 定时统计 | OpenTelemetry/Activity | ASP.NET Core 生态 |
| **错误表达** | ApiCode + 数据库异常脱敏，可切换 UseHttpStatus | StatusCode + RpcException | HTTP 状态码 + JSON 错误体 |
| **重试** | IRetryPolicy（可插拔），401 不计入重试 | 需自行实现 | 需自行实现 |
| **框架兼容** | net45 ~ net10.0 | .NET Core 3.1+ | .NET 6+ |

### NewLife.Remoting 指令下发架构要点

#### 核心架构

```
PublishAsync(code, cmd)
  → Bus.PublishAsync("code#json")          ← 事件总线全量广播（Redis/星尘/内存）
  → 全部实例 OnMessage
    → Get(code) → 有 session → HandleAsync ← 有设备WS的实例下发
    → Get(code) → null → 忽略              ← 其他实例自动忽略
```

- **不依赖独立路由表**：事件总线广播天然解决集群路由问题。消息一经发布，所有实例都能收到，持有设备 WebSocket 会话的实例负责下发，其余实例自然忽略。
- **多通道冗余**：WebSocket 主通道（实时全双工）> SSE 降级通道（HTTP 单向推送）> Ping 心跳搭载（最终兜底），确保指令可达。
- **响应关联**：设备执行完毕后通过 CommandReply HTTP 上报结果，经事件总线广播回到所有实例，发起调用的实例通过 CommandId 匹配 TaskCompletionSource 完成等待。

#### 与竞品的关键差异

| 竞品做法 | NewLife.Remoting 做法 | 理由 |
|:---------|:---------------------|:------|
| 独立路由表（EMQX Mria、ThingsBoard） | **事件总线全量广播** | 消息总线广播已保证所有实例收到消息，无需额外路由表。广播是传输手段而非浪费——设备连接是动态的，广播确保消息不错过目标实例 |
| MQTT QoS 持久化队列 | **在线即时下发 + 心跳搭载** | 保持架构轻量，设备离线时命令不下发，重连后通过心跳扫描积压命令。业务方可按需注入持久化实现 |
| Protobuf 强契约 | **JSON + Packet 二进制直通** | 降低接入门槛，便于调试；热点路径可用二进制 Packet 绕开序列化 |

## 4. 结论与建议

### 优势领域

| 领域 | 说明 |
|:-----|------|
| **统一传输协议** | 单一框架同时覆盖 TCP/UDP/HTTP/WS，减少技术栈碎片 |
| **全框架兼容** | net45 ~ net10.0，是唯一支持 .NET Framework 的现代 RPC 框架 |
| **流式调用** | Server-Streaming + SSE 双模式，浏览器可原生消费 |
| **指令下发架构** | 事件总线广播 + WS/SSE/Ping 多通道冗余，无需独立路由表 |
| **IoT 一站式** | 客户端基类 + 自动更新 + 指令下发 + 设备管理，开箱即用 |

### 差距领域

| 领域 | 差距 | 追赶优先级 |
|:-----|:-----|:----------:|
| **跨语言互操作** | JSON 序列化效率低于 Protobuf，无 .proto 契约 | 低（聚焦 .NET 生态） |
| **离线指令持久化** | 设备离线时不下发，无持久化队列 | 中（可按需注入） |
| **社区生态** | 相比 gRPC/ASP.NET Core，文档和社区资源较少 | 中（持续建设中） |

### 选型建议

| 需求 | 推荐 |
|:-----|:-----|
| 统一 RPC + HTTP + WS，.NET 全框架兼容 | **NewLife.Remoting** |
| 跨语言微服务通信 | **gRPC** |
| IoT 设备接入，需要指令下发/自动更新 | **NewLife.Remoting + Extensions** |
| 开放 REST API，需 Swagger/OpenAPI | **Minimal APIs** |
| 海量 MQTT 设备接入 | **EMQX**（MQTT Broker）+ **NewLife.Remoting**（业务层） |
