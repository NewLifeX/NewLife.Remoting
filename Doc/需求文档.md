# NewLife.Remoting 需求文档

> 版本：v3.8 | 日期：2026-07-23
>
> 本文档描述 NewLife.Remoting 的愿景、核心目标和功能方向。完成状态在[功能清单](功能清单.md)中追踪，详细设计在[架构设计](架构设计.md)中展开。

## 1. 背景与愿景

### 1.1 系统定位

NewLife.Remoting 是 NewLife 技术栈的统一远程通信基础设施，由两个 NuGet 包组成：

| 包 | 定位 | 覆盖 |
|:---|:-----|:------|
| **NewLife.Remoting** | 核心通信库 | RPC 客户端/服务端、协议编解码、连接管理、流式调用、客户端基类、指令下发管线、自动更新 |
| **NewLife.Remoting.Extensions** | 扩展库 | 服务端控制器基类（BaseController/BaseDeviceController/BaseOAuthController）、设备管理服务、Token 鉴权、命令响应总线 |

### 1.2 愿景

使 .NET 开发者能在 net45 ~ net10.0 单一生态内完成全部远程通信需求——RPC（二进制高性能）、HTTP/REST、WebSocket 长连接、SRMP 远程消息协议，同时为 IoT 场景提供指令下发、设备管理和自动更新的一站式解决方案。

## 2. 核心目标

| 编号 | 目标 | 层级 | 描述 |
|:----|:-----|:----:|------|
| RPC | RPC 核心框架 | 核心层 | SRMP 协议、编解码、流水线、客户端/服务端、Cluster 连接管理、元数据传递、重试策略 |
| STRM | 流式调用 | 核心层 | Server-Streaming：`IAsyncEnumerable<T>` 逐条推送，SRMP + SSE 双协议 |
| CLNT | 客户端基类 | 核心层 | 登录/心跳/升级/指令/事件，WS/SSE 双通道 |
| UPGD | 自动更新 | 核心层 | 检测→下载→校验→解压→覆盖→重启 |
| SRV | 服务端基础 | 扩展层 | ApiController/BaseController/BaseDeviceController、TokenService、IDeviceService |
| CMD | 指令下发 | 扩展层 | SessionManager + 事件总线广播 + WS/SSE/Ping 多通道 |

## 3. 功能需求

### 3.1 RPC — RPC 核心框架

> 排序原则：协议消息 → 编解码 → 内存管理 → 元数据模型 → 处理流水线 → 连接管理 → 客户端/服务端 → 会话扩展 → 元数据传递 → 重试策略
>
> 元数据传递（RPC-11~RPC-16）和重试策略（RPC-17~RPC-20）是 ApiClient 的内置小功能，作为 RPC 子项纳入。

- **RPC-1 SRMP 协议消息模型**：SRMP 协议消息模型 Flag/Seq/Length/Payload，核心通信协议基础
- **RPC-2 Encoder 编解码体系**：`IEncoder` 接口体系，JSON 默认编码（JsonEncoder），支持 HTTP 编码（HttpEncoder），二进制 Packet 直通，支持替换编码器
- **RPC-3 内存管理**：`IOwnerPacket`/`OwnerPacket` 零拷贝链式传递，ArrayPool 池化，级联释放
- **RPC-4 ApiAction 动作元数据**：反射预编译加速，IsPacket/IsAccessor/IsStreaming 属性检测，支持二进制参数 Packet、流式返回 IAsyncEnumerable
- **RPC-5 ApiManager 动作管理**：控制器注册与发现，类型注册和实例注册，名称到动作的映射
- **RPC-6 ApiHandler 处理器**：请求路由到 Controller，ControllerContext 构建与生命周期管理，参数绑定与解码
- **RPC-7 Cluster 连接管理**：单连接（`ClientSingleCluster`）故障转移 / 连接池（`ClientPoolCluster`）负载均衡两种模式，被 ApiClient 用于连接管理
- **RPC-8 ApiClient 客户端**：TCP/UDP/HTTP 多协议客户端，通过 Cluster 管理连接、自动故障转移、401 自动重登，Headers/Token 自动注入
- **RPC-9 ApiServer 服务端**：多协议监听，请求处理流水线（Process），支持 Controller 和委托模式，流式推送
- **RPC-10 会话级别 Encoder**：每个会话可通过 `session["Encoder"]` 独立设置 Encoder，覆盖全局编码器
- **RPC-11 ApiClient.Headers 属性**：`IDictionary<String, String?>` 属性，每次调用自动注入
- **RPC-12 InvokeAsync 自动注入 Headers**：调用时将 Headers 合并到参数字典顶级 key
- **RPC-13 InvokeStreamAsync 自动注入 Headers**：流式调用同样自动注入 Headers
- **RPC-14 TraceId 自动注入**：`DefaultSpan.Current?.TraceId` 非空时自动注入到 Headers["TraceId"]
- **RPC-15 服务端提取 Headers**：Headers 合并到参数字典后以方法参数形式可达 Controller，挂载到 `ctx.Parameters` 中
- **RPC-16 Controller 读取 Headers**：Controller 方法参数直接接收 Headers 中的字段，或通过 `ControllerContext.Current.Parameters` 访问
- **RPC-17 IRetryPolicy 接口**：`ShouldRetry(Exception, Int32 attempt, out TimeSpan delay, out Boolean refreshClient)` 按异常类型和重试次数决策
- **RPC-18 MaxRetries 配置**：`ApiClient.MaxRetries` 不含首次尝试，0 表示禁用
- **RPC-19 指数退避延迟**：策略实现通过 `delay` 输出参数自行控制退避算法
- **RPC-20 连接切换**：`refreshClient` 为 true 时 Cluster.Return 归还当前连接再重新获取

### 3.2 STRM — 流式调用 (Server-Streaming)

> 排序原则：协议标志 → 元数据检测 → 编码器支持 → 服务端处理 → SSE 兼容 → 生命周期 → 客户端 API → 代理

- **STRM-1 SRMP 协议 Streaming Flag**：Flag 字节 bit5(Streaming) + bit4(EndOfStream) 标记流式帧，核心协议扩展
- **STRM-2 ApiAction.IsStreaming 检测**：反射检测返回类型 `IAsyncEnumerable<T>` 自动标记流式
- **STRM-3 Controller 流式检测**：Process 中通过接口检测 IAsyncEnumerable 自动启用流式，支持值类型和引用类型
- **STRM-4 Encoder 流式块编码**：`EncoderBase.EncodeStreamChunk` 预计算长度，SpanWriter 合并写入
- **STRM-5 ProcessStream 服务端**：泛型 Helper `SendStreamItems<T>` 处理值类型/引用类型流，异常时发送错误帧
- **STRM-6 HTTP SSE 兼容**：输出 `text/event-stream`，浏览器和 curl 可调试，基于 `HttpEncoder.CreateStreamResponse`
- **STRM-7 结束标记帧**：空 action + 空 data 标记流结束
- **STRM-8 客户端断开自动停止**：服务端通过 CancellationToken（链接 Session.Disposed 轮询）感知断开，自动停止枚举
- **STRM-9 InvokeStreamAsync 客户端**：返回 `IAsyncEnumerable<T>`，SemaphoreSlim + ConcurrentQueue 收集帧，支持 CancellationToken 中途取消
- **STRM-10 ClientBase 代理**：ClientBase 代理底层 RPC 客户端的 InvokeStreamAsync

### 3.3 CLNT — 客户端基类

> 依赖 RPC 核心框架（ApiClient），调用 UPGD Upgrade 实现自动更新
> 排序原则：核心生命周期 → 通道层 → 通信层 → 命令委托注册 → 命令处理 → 业务功能

- **CLNT-1 核心生命周期**：打开/关闭/重连，Features 标志位（Login/Logout/Ping/Upgrade/Notify/CommandReply/PostEvent）控制功能开关
- **CLNT-2 WebSocket 通道**：WS 连接管理、自动重连、心跳保活（`WsChannel`）
- **CLNT-3 SSE 通道**：SSE 降级通道，WS 不可用时自动启用（`SseChannel`）
- **CLNT-4 登录/注销**：自动登录、自动重登、Token 管理
- **CLNT-5 心跳 Ping**：定时心跳，搭载积压命令下发，更新在线状态
- **CLNT-6 Commands 委托注册**：`Commands` 字典注册命令名到处理委托，被 CLNT-7 命令接收与执行管线使用，支持同步/异步处理器
- **CLNT-7 命令接收与执行**：`ReceiveCommand` → `OnReceiveCommand` → `ExecuteCommand` 委托分发管线，通过 `Commands` 字典匹配处理委托
- **CLNT-8 事件上报**：批量上报设备事件（`PostEvents`）
- **CLNT-9 流式调用代理**：代理底层 RPC 客户端的 `InvokeStreamAsync`
- **CLNT-10 自动升级**：调用 UPGD `Upgrade` 执行全链路更新：检测→下载→校验→解压→覆盖→重启

### 3.4 UPGD — 自动更新

> 独立文件操作工具类，提供下载/解压/覆盖/回滚/重启全链路能力，被 CLNT 客户端基类的自动升级功能调用

- **UPGD-1 文件操作**：下载/解压/文件覆盖/回滚，先改名再拷贝策略
- **UPGD-2 版本去重**：`_lastVersion` 避免重复更新同一版本
- **UPGD-3 下载与哈希校验**：HTTP 下载 .zip，MD5/SHA256 校验
- **UPGD-4 解压与覆盖**：安全替换运行中文件，Linux sync 强制落盘
- **UPGD-5 跨平台重启**：Windows/Linux/OSX 拉起新进程，支持 systemd
- **UPGD-6 回滚保护**：更新失败时自动恢复被移动的旧文件

### 3.5 SRV — 服务端基础

> 依赖 RPC 核心框架（ApiServer/Encoder）和 CMD 指令下发（ISessionManager）
> 排序原则：独立诊断控制器 → 服务接口/实现 → 令牌服务 → 控制器基类 → OAuth 控制器 → 设备控制器

- **SRV-1 ApiController 诊断控制器**：内置 All/Info/GetIps 等诊断端点，运行信息查询
- **SRV-2 IDeviceService 设备服务接口**：设备 CRUD、登录/注销/心跳/命令/事件/升级/令牌验证
- **SRV-3 DefaultDeviceService 设备服务默认实现**：`IDeviceService` 抽象基类，提供默认的登录/心跳/命令/事件/升级逻辑，支持派生类重写
- **SRV-4 TokenService 令牌服务**：JWT 令牌编码/解码/验证，区分应用令牌和设备令牌，自动续期
- **SRV-5 BaseController 基础控制器**：令牌解码与验证、设备上下文（DeviceContext）管理、异常收敛处理
- **SRV-6 BaseOAuthController OAuth 控制器**：令牌颁发与刷新，OAuth 认证流程
- **SRV-7 BaseDeviceController 设备控制器基类**：HTTP 端点 Login/Logout/Ping/Upgrade/Notify/NotifySSE/CommandReply/SendCommand/PostEvents，设备生命周期管理。内部创建 WsCommandSession/SseCommandSession，使用 CMD ISessionManager 进行命令下发与响应等待

### 3.6 CMD — 指令下发

> 依赖 RPC 核心框架。SessionManager/CommandSession 为核心层独立管线，WsCommandSession/SseCommandSession 为扩展层实现，被 SRV 服务端基础的 BaseDeviceController 调用

- **CMD-1 SessionManager 会话管理器**：会话 CRUD（Add/Remove/Get）、命令下发（PublishAsync）、响应等待（WaitResponseAsync）、响应广播（PublishResponseAsync）完整闭环
- **CMD-2 事件总线命令发布**：`PublishAsync` 通过 EventBus 广播 `code#json`，支持多实例集群
- **CMD-3 响应等待与回调注册**：TCS 回调注册到 `_callbacks` 字典，超时自动清理，支持 CancellationToken
- **CMD-4 响应广播管线**：设备 CommandReply 通过 EventBus ResponseTopic 广播到所有实例，`OnCommandResponse` 按 CommandId 匹配回调
- **CMD-5 CommandSession 基类**：通用会话基类，管理 Code/Active/SetOnline，派生类实现 HandleAsync
- **CMD-6 WsCommandSession**：WebSocket 长连接命令下发，支持 Ping/Pong 心跳、事件广播分发
- **CMD-7 SseCommandSession**：SSE 轻量降级通道，WS 不可用时的替代方案，支持心跳注释防止代理超时
- **CMD-8 向后兼容降级**：SessionManager 事件总线模式，优先使用 Redis 跨进程总线（通过 IEventBusFactory），降级到内存总线；CommandReply 通过 EventBus ResponseTopic 广播到所有实例

## 4. 🧊 暂缓清单

| 功能 | 暂缓理由 | 解冻前提 |
|:-----|:---------|:----------|
| 客户端流式 (Client Streaming) | Server-Streaming 已覆盖 IoT 推送场景 | 出现明确的客户端→服务端流式需求 |
| 双向流式 (Bidirectional Streaming) | WebSocket 长连接已提供类似能力 | WebSocket 不满足的场景出现 |
| Protobuf 支持 | 保持 JSON + Packet 二进制直通，避免引入新序列化 | 跨语言互操作成为强需求 |
| UDP 流式调用 | UDP 无连接，流式语义不适合 | 无 |
| 组件图/时序图自动生成 | Mermaid 文本图已满足文档需求 | 文档规模扩大需要自动生成 |

## 5. 不做什么

- 不改 `IMessage` / `IActionFilter` / `ICluster` 等 NewLife.Core 基础接口
- 不新增外部 NuGet 依赖
- 不替换现有 JSON 序列化方案
- 不引入 gRPC 或 SignalR 作为依赖
