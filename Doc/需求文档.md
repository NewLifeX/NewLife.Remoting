# NewLife.Remoting 需求文档

> 版本：v3.8 | 日期：2026-07-23
>
> 本文档描述 NewLife.Remoting 的愿景、核心目标和功能方向。完成状态在[功能清单](功能清单.md)中追踪，详细设计在[架构设计](架构设计.md)中展开。

## 1. 背景与愿景

### 1.1 系统定位

NewLife.Remoting 是 NewLife 技术栈的统一远程通信基础设施，由两个 NuGet 包组成：

| 包 | 定位 | 覆盖 |
|:---|:-----|:------|
| **NewLife.Remoting** | 核心通信库 | RPC 客户端/服务端、协议编解码、连接管理、流式调用、客户端基类、指令下发管线、自动更新 |
| **NewLife.Remoting.Extensions** | 扩展库 | 服务端控制器基类（BaseController/BaseDeviceController/BaseOAuthController）、设备管理服务、Token 鉴权、命令响应总线 |

### 1.2 愿景

使 .NET 开发者能在 net45 ~ net10.0 单一生态内完成全部远程通信需求——RPC（二进制高性能）、HTTP/REST、WebSocket 长连接、SRMP 远程消息协议，同时为 IoT 场景提供指令下发、设备管理和自动更新的一站式解决方案。

## 2. 核心目标

| 编号 | 目标 | 层级 | 描述 |
|:----|:-----|:----:|------|
| RPC | RPC 核心框架 | 核心层 | 多协议远程调用、自动编解码、高可用连接管理、可配置重试、请求路由与参数绑定 |
| STRM | 流式调用 | 核心层 | 服务端向客户端逐条推送数据，支持 SRMP 和 SSE 双协议 |
| CLNT | 客户端基类 | 核心层 | 客户端一站式生命周期：登录、心跳、升级、指令、事件，WS/SSE 双通道 |
| UPGD | 自动更新 | 核心层 | 检测、下载、校验、解压、覆盖、重启全链路自动更新 |
| SRV | 服务端基础 | 扩展层 | 诊断端点、令牌鉴权、OAuth 认证、设备生命周期管理控制器 |
| CMD | 指令下发 | 扩展层 | 云端到设备的指令下发、响应等待、集群广播 |

## 3. 功能需求

### 3.1 RPC — RPC 核心框架

> 排序原则：协议消息 → 编解码 → 内存管理 → 元数据模型 → 处理流水线 → 连接管理 → 客户端/服务端 → 会话扩展 → 元数据传递 → 重试策略

- **RPC-1 自定义协议帧**：应用可通过自定义协议帧高效完成远程调用，支持粘拆包、双向消息、单向通知
- **RPC-2 可扩展编解码**：支持 JSON 和二进制两种序列化方式，可按需替换编码器，兼顾调试便利性和传输效率
- **RPC-3 零拷贝内存管理**：大吞吐场景下零拷贝传递数据，避免频繁内存分配和拷贝
- **RPC-4 控制器元数据识别**：框架自动识别控制器方法的参数类型和返回类型，支持二进制参数和流式返回
- **RPC-5 控制器自动注册**：控制器自动注册，开发者只需编写普通类即可暴露远程服务
- **RPC-6 请求路由与参数绑定**：请求自动路由到控制器方法，框架自动完成参数绑定和生命周期管理
- **RPC-7 高可用连接管理**：客户端连接支持单节点故障转移和连接池负载均衡，保障服务可用性
- **RPC-8 多协议客户端**：客户端通过统一 API 连接远程服务，支持自动重连、故障转移和身份凭证注入
- **RPC-9 多协议服务端**：服务端同时监听多种传输协议，支持控制器和委托两种处理模式
- **RPC-10 会话级编码隔离**：支持为不同客户端连接独立设置编码方式，适应混合协议场景
- **RPC-11 调用上下文透传**：调用上下文（如链路追踪 ID、租户标识）自动透传到服务端，无需显式传参
- **RPC-12 可配置重试策略**：业务层可自定义重试策略，按异常类型和次数决定是否重试
- **RPC-13 重试次数上限**：控制最大重试次数，防止无限重试导致资源耗尽
- **RPC-14 指数退避**：重试间隔自动递增，避免重试风暴
- **RPC-15 重试切换节点**：重试时自动切换到健康节点，提高成功率

### 3.2 STRM — 流式调用 (Server-Streaming)

> 排序原则：协议标志 → 元数据检测 → 编码器支持 → 服务端处理 → SSE 兼容 → 生命周期 → 客户端 API → 代理

- **STRM-1 流式通信协议**：支持服务端向客户端逐条推送数据的流式通信模式
- **STRM-2 自动流式检测**：控制器方法返回流式类型时自动启用推送模式
- **STRM-3 值类型与引用类型兼容**：流式推送兼容值和引用类型
- **STRM-4 流式数据分块编码**：流式数据自动分块编码传输，客户端逐条接收
- **STRM-5 服务端持续推送**：服务端可向客户端持续推送数据流，支持异常中断通知
- **STRM-6 HTTP SSE 兼容**：流式推送兼容标准 SSE 协议，浏览器可直接消费
- **STRM-7 推送完成通知**：数据流推送完毕自动通知客户端
- **STRM-8 客户端断开自动停止**：客户端断开连接时服务端自动停止推送，避免资源浪费
- **STRM-9 客户端逐条消费**：客户端通过 `await foreach` 逐条消费服务端推送的数据
- **STRM-10 客户端基类封装**：客户端基类封装流式调用接口，开箱即用

### 3.3 CLNT — 客户端基类

> 提供开箱即用的客户端生命周期管理，调用 UPGD Upgrade 实现自动更新

- **CLNT-1 生命周期管理**：提供开箱即用的客户端生命周期管理：打开、关闭、自动重连，功能开关可配置
- **CLNT-2 WebSocket 长连接**：支持 WebSocket 长连接，实现服务端到客户端的实时推送
- **CLNT-3 SSE 降级通道**：WebSocket 不可用时自动降级为 SSE 通道，保证指令可达
- **CLNT-4 自动登录与令牌管理**：支持自动登录和令牌管理，令牌过期自动重新认证
- **CLNT-5 定时心跳保活**：定时心跳保持在线状态，同时搭载积压命令下发
- **CLNT-6 命令委托注册**：支持按命令名注册处理委托，灵活响应不同类型的指令
- **CLNT-7 命令接收与分发**：收到命令后自动分发到已注册的处理方法
- **CLNT-8 批量事件上报**：支持批量上报设备事件，减少网络开销
- **CLNT-9 流式调用代理**：支持流式调用，逐条接收服务端推送的数据
- **CLNT-10 一键自动升级**：检测新版本、下载、校验、解压、覆盖、重启全链路

### 3.4 UPGD — 自动更新

> 提供下载、解压、覆盖、回滚、重启完整的自动更新链路，被 CLNT 客户端基类的自动升级功能调用

- **UPGD-1 全链路更新**：支持从远程下载升级包、解压覆盖旧文件，失败时自动回滚
- **UPGD-2 版本去重**：同一版本只更新一次，避免重复下载
- **UPGD-3 完整性校验**：下载后自动校验文件完整性，防止损坏
- **UPGD-4 安全覆盖**：安全替换正在运行的文件，支持 Linux 系统
- **UPGD-5 跨平台重启**：更新完成后自动重启进程，支持 Windows/Linux/OSX
- **UPGD-6 回滚保护**：更新失败时自动恢复旧版本，保证系统不崩溃

### 3.5 SRV — 服务端基础

> 提供服务端控制器基类和设备管理服务，依赖 RPC 核心框架和 CMD 指令下发

- **SRV-1 诊断端点**：内置诊断端点，查询服务运行状态和配置信息
- **SRV-2 设备服务接口**：提供设备生命周期管理接口：注册、登录、心跳、注销
- **SRV-3 默认设备服务**：内置设备服务默认实现，开箱即用，也可自定义业务逻辑
- **SRV-4 令牌服务**：支持 JWT 令牌颁发与验证，区分应用和设备两种身份
- **SRV-5 统一鉴权与控制**：提供统一的令牌鉴权、设备上下文管理和异常处理
- **SRV-6 OAuth 认证**：支持 OAuth 标准认证流程，令牌颁发与刷新
- **SRV-7 设备全生命周期管理**：覆盖设备完整生命周期：登录认证 → 心跳保活 → 指令下发 → 升级管理 → 事件上报

### 3.6 CMD — 指令下发

> 提供云端到设备的指令下发、执行和结果回传完整管线，依赖 RPC 核心框架

- **CMD-1 会话管理**：管理设备 WebSocket 会话，支持指令下发和响应等待
- **CMD-2 事件总线广播**：指令通过事件总线广播到所有服务实例，持有设备会话的实例负责下发
- **CMD-3 同步等待响应**：支持同步等待设备响应，超时自动清理，避免内存泄漏
- **CMD-4 响应回传**：设备响应通过事件总线回传，发起实例通过指令 ID 匹配回调
- **CMD-5 通用会话基类**：提供通用会话基类，管理设备在线状态
- **CMD-6 WebSocket 指令通道**：WebSocket 长连接通道，支持心跳保活和事件推送
- **CMD-7 SSE 降级通道**：SSE 降级通道，WebSocket 不可用时保障指令可达
- **CMD-8 集群兼容**：优先使用 Redis 事件总线支持跨进程集群，自动降级到内存总线

## 4. 🧊 暂缓清单

| 功能 | 暂缓理由 | 解冻前提 |
|:-----|:---------|:----------|
| 客户端流式 (Client Streaming) | Server-Streaming 已覆盖 IoT 推送场景 | 出现明确的客户端→服务端流式需求 |
| 双向流式 (Bidirectional Streaming) | WebSocket 长连接已提供类似能力 | WebSocket 不满足的场景出现 |
| Protobuf 支持 | 保持 JSON + Packet 二进制直通，避免引入新序列化 | 跨语言互操作成为强需求 |
| UDP 流式调用 | UDP 无连接，流式语义不适合 | 无 |
| 组件图/时序图自动生成 | Mermaid 文本图已满足文档需求 | 文档规模扩大需要自动生成 |

## 5. 不做什么

- 不改 `IMessage` / `IActionFilter` / `ICluster` 等 NewLife.Core 基础接口
- 不新增外部 NuGet 依赖
- 不替换现有 JSON 序列化方案
- 不引入 gRPC 或 SignalR 作为依赖
