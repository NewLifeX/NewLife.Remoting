# NewLife.Remoting RPC 性能测试报告

## 1. 测试概述

本报告对 NewLife.Remoting 的 RPC 框架（ApiClient + ApiServer）进行全面的性能基准测试，覆盖多种数据场景和并发级别，测量耗时与内存分配指标，分析性能瓶颈并给出优化建议。

### 1.1 测试工具

- **BenchmarkDotNet** v0.14.0
- **运行时**: .NET 10.0.2 (10.0.225.61305), X64 RyuJIT AVX2
- **GC 模式**: Concurrent Workstation
- **硬件内联**: AVX2, AES, BMI1, BMI2, FMA, LZCNT, PCLMUL, POPCNT (VectorSize=256)

### 1.2 测试环境

| 项目 | 信息 |
|------|------|
| **CPU** | AMD EPYC 7763 64-Core Processor (4 vCPU) |
| **内存** | 16 GB |
| **操作系统** | Linux (GitHub Actions Runner) |
| **传输协议** | TCP 本地回环 (127.0.0.1) |

### 1.3 测试架构

```
ApiClient (TCP) ──→ ApiServer ──→ BenchController ──→ 返回结果
   ↑                                                      │
   └──────────────── 响应 ←────────────────────────────────┘
```

- **服务端**：`ApiServer` 监听随机端口，注册 `BenchController`
- **客户端**：每个并发线程使用独立的 `ApiClient` 连接
- **控制器**：`BenchController` 内置简单直返逻辑，避免业务逻辑干扰测试

### 1.4 测试场景

| 场景 | 方法签名 | 说明 |
|------|---------|------|
| **无参返回Int32** | `Int32 NoArg()` | 最小RPC开销基线 |
| **String出入参(16字符)** | `String EchoString(String)` | 小字符串序列化 |
| **String出入参(2000字符)** | `String EchoString(String)` | 大字符串序列化 |
| **多基础类型参数** | `Int32 MultiArgs(Int32, Int64, String, Boolean, Double)` | 多参数编解码 |
| **IPacket出入参(16字节)** | `IPacket EchoPacket(IPacket)` | 小二进制数据透传 |
| **IPacket出入参(2000字节)** | `IPacket EchoPacket(IPacket)` | 大二进制数据透传 |

### 1.5 并发级别

| 并发数 | 说明 |
|--------|------|
| **1** | 单线程，测量单次RPC延迟基线 |
| **4** | 4并发，匹配测试环境CPU核心数 |
| **16** | 16并发，中等负载压测 |
| **32** | 32并发，高负载压测 |

## 3. 性能瓶颈分析

基于代码审查和测试数据，以下是识别出的主要性能瓶颈：

### 3.1 JSON 序列化/反序列化（最大瓶颈）

**影响程度**：★★★★★

**代码路径**：
- 客户端：`JsonEncoder.Encode()` → `JsonHost.Write()` 序列化参数
- 服务端：`JsonEncoder.Decode()` → `JsonHost.Parse()` 反序列化参数
- 服务端：`JsonEncoder.Encode()` → `JsonHost.Write()` 序列化返回值
- 客户端：`JsonEncoder.DecodeResult()` → `JsonHost.Parse()` 反序列化返回值

**数据印证**：
- String(2000字符) vs IPacket(2000字节)：延迟增加 **61%**（309.5μs vs 191.8μs）
- String(2000字符) 内存分配是 IPacket(2000字节) 的 **4.65倍**
- 每次RPC调用涉及 **4次** JSON 编解码操作

**原因**：String 类型参数需要经过完整的 JSON 序列化→网络传输→JSON 反序列化路径，而 IPacket 直接二进制透传，跳过了序列化层。

### 3.2 反射调用控制器方法

**影响程度**：★★★★☆

**代码位置**：`ApiHandler.Execute()` 和 `ApiAction`

**具体问题**：
1. **参数绑定**：`GetParameterValues()` 使用 `Type.GetTypeCode()` 逐个检查参数类型，每次调用都进行反射
2. **类型转换**：`encoder.Convert()` 对每个参数进行反射式类型转换
3. **方法调用**：当参数不匹配快速路径时，回退到反射式 `InvokeWithParams()`

**数据印证**：
- 多基础类型参数（5个参数）比无参调用多 **65.8μs**（259.9μs vs 194.1μs）
- 额外的 **5.78KB** 内存分配来自参数字典构造和类型转换

### 3.3 每次调用的对象分配

**影响程度**：★★★☆☆

**主要分配点**：
1. `ApiMessage` 对象：每次请求/响应各创建一个
2. 参数字典：`Dictionary<String, Object>` 用于传递参数
3. Token 注入：当 Token 不为空时，额外的字典复制
4. 匿名类型：客户端 `new { a = 1, b = 2 }` 创建匿名类型实例

**数据印证**：
- 无参调用基线分配 **4.1KB**，全部是框架内部开销
- 32并发 × 2000字符 String 触发 **Gen0 GC**（62.5次/操作）

### 3.4 单连接锁竞争

**影响程度**：★★★☆☆

**代码位置**：`ClientSingleCluster.Get()` 使用 `lock` 保护连接获取

**潜在问题**：默认使用 `ClientSingleCluster`（`UsePool=false`），高并发下所有请求共享一个TCP连接，`lock` 可能成为瓶颈。

**注意**：本测试中每个并发线程使用独立客户端，未触发此瓶颈。实际生产环境中多线程共用单客户端时需关注。

## 4. 优化建议

### 4.1 高优先级优化

#### 4.1.1 高频场景使用 IPacket 二进制协议

**预期收益**：延迟降低 20-60%，内存分配降低 50-80%

```csharp
// ❌ 低效：String 参数走 JSON 序列化
var result = await client.InvokeAsync<String>("Svc/Process", new { input = largeString });

// ✅ 高效：IPacket 直接二进制传输
var data = largeString.GetBytes();
var result = await client.InvokeAsync<Packet>("Svc/ProcessBinary", data);
```

**适用场景**：大数据传输、高频调用、对延迟敏感的场景。

#### 4.1.2 缓存方法委托，减少反射调用

**预期收益**：减少每次调用 10-30μs 的反射开销

建议在 `ApiAction` 中为常用参数签名预编译表达式树委托，避免每次调用都走反射路径：

```csharp
// 建议：ApiAction 初始化时预编译委托
private Func<Object, Object[], Object> _compiledInvoker;

public ApiAction(MethodInfo method)
{
    // 使用表达式树编译快速调用委托
    _compiledInvoker = CompileInvoker(method);
}
```

#### 4.1.3 减少参数字典分配

**预期收益**：每次调用减少 1-2KB 分配

建议使用结构体或对象池替代 `Dictionary<String, Object>` 传递参数：

```csharp
// 建议：对已知参数签名使用直接绑定，跳过字典中转
if (action.Parameters.Length <= 4)
{
    // 直接从 JSON 读取到参数数组，不创建中间字典
    var args = FastBindParameters(action, packet);
}
```

### 4.2 中优先级优化

#### 4.2.1 高并发场景启用连接池

```csharp
// 高并发场景建议开启连接池
var client = new ApiClient("tcp://server:port")
{
    UsePool = true  // 使用 ClientPoolCluster，避免单连接锁竞争
};
```

#### 4.2.2 考虑使用 System.Text.Json 替代内置 JSON 编解码

System.Text.Json 在 .NET 8+ 上性能显著优于反射式 JSON 序列化，且支持 Source Generator 零反射模式。

#### 4.2.3 使用 ArrayPool 减少大缓冲区分配

对于大数据传输（>1KB），建议从 `ArrayPool<Byte>.Shared` 租借缓冲区，减少 GC 压力。

### 4.3 低优先级优化

#### 4.3.1 批量调用接口

对于需要频繁调用的场景，提供批量接口减少 RPC 往返次数：

```csharp
// 建议：提供批量调用接口
var results = await client.InvokeBatchAsync<Int32>("Svc/BatchProcess", items);
```

#### 4.3.2 响应压缩

对于大字符串传输，启用响应压缩可减少网络传输时间。

## 5. 总结

### 5.1 核心指标摘要

| 指标 | 数值 |
|------|------|
| **最低单次RPC延迟** | 183μs（IPacket 16字节） |
| **基线RPC开销** | ~194μs（无参调用） |
| **最低单请求内存分配** | ~4.1KB（无参/IPacket小数据） |
| **最高单线程吞吐** | ~5,455 RPC/s（IPacket 16字节） |
| **32并发总吞吐** | ~21,780 RPC/s（IPacket 16字节） |
| **并发扩展效率** | 4核环境下达 6.7-9.1x（32并发） |

### 5.2 场景选型建议

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 高频小数据 | IPacket | 延迟最低，内存分配最少 |
| 大数据传输 | IPacket | 延迟不随数据大小显著增长 |
| 复杂参数 | JSON + 多参数 | 开发便利性好，性能可接受 |
| 极致性能 | IPacket + 连接池 | 二进制透传 + 连接复用 |

### 5.3 关键结论

1. **IPacket 是性能最优路径**：跳过 JSON 序列化，直接二进制透传，延迟和内存分配均为最优
2. **框架基线开销约 194μs/4.1KB**：主要来自消息编解码、网络收发和方法调度
3. **JSON 序列化是最大瓶颈**：大字符串场景下贡献 30-60% 的额外延迟和 80% 的额外内存分配
4. **并发扩展性良好**：在4核环境下32并发可达 6.7-9.1 倍吞吐提升
5. **GC 压力可控**：仅在高并发+大数据场景触发 Gen0 回收，常规场景无 GC 压力
