# ZeroServer 指令下发与响应总线端到端分析

> 场景：Cube 管理后台「节点在线」表，选中在线节点 → 点击「检查更新」→ 同步等待后返回逐设备执行结果。

---

## 1. 背景

原 `NodeOnlineController.CheckUpgrade` 使用 fire-and-forget 的 `SendCommand(IDeviceModel, CommandModel, ct)`，返回 `Task<Int32>` 后立即刷新页面，设备侧的 `CommandReply` 回复无人消费。

重构后，响应等待下沉到 `SessionManager.PublishAsync` 内部，`NodeOnlineController` 只需调用 `SendCommand(device, cmd, 10, ct)` 即可同步等待设备回复。

---

## 2. 最终架构

### 调用链

```
Admin 点击「检查更新」
  │
  ▼
NodeOnlineController.CheckUpgrade()
  └─► _deviceService.SendCommand(online.Node, cmd, 10, ct)     ← Task<CommandReplyModel?>
        │
        ├─► DefaultDeviceService.PrepareCommand
        │      ├─ WriteHistory
        │      ├─ command.Code / SenderNodeId
        │      └─ SaveCommand → 持久化
        │
        └─► sessionManager.PublishAsync(code, cmd, null, 10, ct)   ← 响应等待下沉到此处
              │
              ├─► Init() → 延迟初始化事件总线
              ├─► Bus.PublishAsync("code#json")                     ← 📤 命令总线：Topic="Commands"
              │     └─► OnMessage → Get(code) → session.HandleAsync → SSE push → 客户端
              │
              ├─ [timeout <= 0] → return null                      ← fire-and-forget 兼容
              │
              └─ [timeout > 0] → 等待响应                            ← 📥 响应总线：Topic="CommandReplies"
                    ├─► ICommandResponseBus.WaitResponseAsync(id, 10, ct)
                    │     └─ EventBus["CommandReplies"].Subscribe → TCS → return reply
                    │
                    └─► 降级: ICacheProvider.GetQueue("cmdreply:{id}").TakeOneAsync(10, ct)
                          └─ 旧 Redis 队列兼容

        │
        ├─► reply != null → 校验状态（错误/取消 → throw）
        └─► return reply → Controller 拿到 CommandReplyModel
```

### 客户端侧（不受影响）

```
SSE → ClientBase.ReceiveCommand("node/upgrade")
  ├─ ExecuteCommand → OnCommand
  │    └─ Commands["node/upgrade"]  ← OrdinalIgnoreCase 匹配 "Node/Upgrade"
  │         └─ ReceiveUpgrade(arguments)
  │              └─ Upgrade(channel) → HTTP GET /Node/Upgrade → 下载/安装
  ├─ e.Reply = CommandReplyModel { Status=已完成, Data=... }
  └─ CommandReply(reply) → HTTP POST /Node/CommandReply
        └─ DefaultDeviceService.CommandReply
              └─ _responseBus?.PublishResponseAsync(model)      ← 发布到响应总线
```

### 关键变更摘要

```
Before:
  DefaultDeviceService.SendCommand(device, cmd, ct)              Task<Int32>
    └─ sessionManager.PublishAsync(code, cmd, null, ct)         Task<Int32>
    └─ (等待逻辑在 DefaultDeviceService 中：_responseBus.WaitResponseAsync)

After:
  DefaultDeviceService.SendCommand(device, cmd, timeout=0, ct)   Task<CommandReplyModel?>
    └─ sessionManager.PublishAsync(code, cmd, null, timeout, ct) Task<CommandReplyModel?>
         ├─ Bus.PublishAsync("code#json")                        ← 命令总线
         └─ (timeout>0) responseBus.WaitResponseAsync(id, t, ct) ← 响应总线（内部闭环）
```

## 5. 决策记录

| 决策 | 结论 |
|------|------|
| 响应总线与命令总线共用 EventBus 实例 | ❌ 不共用，各自独立 IEventBus（不同 Topic）；可共用 IEventBusFactory 基础设施 |
| 响应等待放在哪一层 | `SessionManager.PublishAsync` 内闭环，是最底层基础设施 |
| CancellationToken 位置 | 永远作为方法**最后一个参数** |
| 是否新增 SendCommand 重载 | ❌ 不新增，旧方法直接加 `Int32 timeout = 0` 参数，破坏性变更一次性完成 |
| 客户端是否需要注册 node/upgrade | ❌ 不需要。`ClientBase` 已注册 `Node/Upgrade`（`OrdinalIgnoreCase` 匹配） |
| 降级策略 | 优先 `ICommandResponseBus`，未注册则降级到 Redis 队列 `cmdreply:{id}` |
| Cube 弹窗展示结果 | 当前通过 `JsonRefresh` 展示统计文本，详细弹窗需前端 JS 配合（超出本阶段范围） |
